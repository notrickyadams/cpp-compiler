# Experiments (E1/E2/E3/E4/E5)

Every number in the paper's evaluation section is regenerated by the
three commands at the bottom of this file. Raw data is kept as CSV so
the analysis is re-runnable and auditable separately from collection.

## Question

What does the explanation machinery cost? Two mechanisms are always-on
in the shipped compiler and do work even when no diagnostic ever fires:

1. **Trace recording** — every scoped stage method pushes/pops a frame
   (and builds its detail string) so that IF a diagnostic fires, the
   real execution path is available to snapshot.
2. **Provenance** — every emitted IR instruction is stamped with its
   source span; optimizer passes preserve spans and append
   transformation notes.

## Design

Factorial ablation over two compile-time flags:

| config   | recording | provenance | isolates                    |
|----------|-----------|------------|-----------------------------|
| full     | on        | on         | the shipped compiler        |
| noprov   | on        | off        | full−baseline share: trace  |
| notrace  | off       | on         | full−baseline share: prov.  |
| baseline | off       | off        | conventional-compiler floor |

The flags compile the mechanisms OUT (see `TRACE_SCOPE` in
`src/diagnostics/TraceRecorder.hpp` for why a macro rather than a
no-op class: argument expressions must vanish too, or the measurement
understates the cost). Ablated builds are for measurement only —
trace-content and provenance tests fail there by design.

## Workload

Deterministic, seeded, valid programs (no diagnostics fire — this
measures the *always-on* cost, the fair worst case for the
architecture) at 100 / 1,000 / 5,000 / 20,000 lines, generated by
`tools/gen_corpus.py`: many 20–40-statement functions of mixed-depth
integer arithmetic. The compile path measured is `--check`: every
pipeline stage through assembly text, no JSON serialization, no
external `g++` subprocess, and NOTHING printed on success — so the
stopwatch sees the compiler and only the compiler.

**Discarded first campaign (methodological note):** the first E1
collection timed `--json` with output piped to `$null`. Its numbers
were incoherent — ablated configs measuring *slower* than the full
build, orderings flipping between sizes, and a 9-second "baseline"
for a 20k-line file through an O(n) pipeline. Root cause: PowerShell
was consuming megabytes of serialized pipeline state through its
object pipeline, so the measurement was dominated by the harness,
and varied with each config's JSON size. `--check` exists precisely
to remove that channel. The invalid CSVs were deleted, not archived —
they measured PowerShell.

## Protocol

- Binaries built `-O2` (dev default is `-O0 -g`; measuring unoptimised
  string handling would inflate the mechanisms' apparent cost).
- Per (config, size): 3 warmups, then 30 timed runs (`Stopwatch`
  around the call operator), then 5 memory runs (`PeakWorkingSet64`
  read after exit).
- Statistics: median and MAD per cell (wall times on a desktop OS are
  right-skewed; medians resist scheduler/AV spikes). Overhead reported
  as (median_config − median_baseline)/median_baseline with a 95%
  bootstrap CI (10,000 resamples, percentile method, fixed seed).
- Known threats, disclosed in the paper: single machine/OS/toolchain;
  process-startup cost dominates the smallest size (that is why the
  size ladder exists); antivirus jitter (median/MAD mitigate; the
  environment is reported with the results).

## E3 — structural completeness (component presence, not quality)

Nine fixed single-error programs (from the project's 20-case
verification suite) go through this compiler and `g++ -fsyntax-only`
on the same files (the language is a C subset, so the comparison is
on identical inputs). The g++ compared against is the local GCC 6.3.0
(MinGW) — the same toolchain the compiler targets. No newer GCC is
installed on the experiment machine; conclusions are therefore about
THIS baseline, and the paper must not imply comparison against
current GCC/Clang diagnostics without running one. `tools/score_diagnostics.py` detects, by
literal mechanical rules listed in the script, which explanation
components each output contains: location, cause, explanation prose,
fix suggestion, code example, trace, caret. Presence only — no
judgement of wording. The enhanced-message effectiveness literature
is contested (Denny et al. 2014; Pettit et al. 2017), so helpfulness
claims are out of scope without a user study; component presence is
claimable mechanically.

## E4 — cascade behaviour on single-fault mutants

`tools/gen_mutants.py` applies exactly one token-level edit per
mutant to a valid seed (delete token / duplicate token / replace an
identifier USE with an undeclared name — declaration sites excluded,
see the script comment for why). Ground truth: one fault per mutant,
so error diagnostics beyond the first are cascade noise (framing:
Diekmann & Tratt, ECOOP 2020). The scorer reports the full
distribution of error counts per rejected mutant for both compilers;
mutants a compiler accepts (harmless edits) are excluded from its
distribution and reported separately.

Disclosed asymmetry: the accepted sets differ BETWEEN compilers
because the languages differ slightly — C permits empty statements
(a duplicated ';') and nested compound blocks (a duplicated '{'),
this language does not — so the two distributions are not over
identical mutant sets. First-run data: g++ accepted 10/100, ours
0/100. A refinement for the paper is to additionally report the
distribution over the INTERSECTION of both compilers' rejected sets.

Run AFTER the timing campaign (it spawns compilers):

```powershell
python tools\gen_corpus.py --lines 60 --out experiments\corpus\seed60.cpp
python tools\gen_mutants.py --seed-file experiments\corpus\seed60.cpp --count 100 --outdir experiments\mutants
python tools\score_diagnostics.py
```

## E5 — invasiveness

Static measurement, no runtime: `git show --stat 48b5054` partitioned
by mechanism gives lines-added for trace recording vs provenance;
`wc -l` gives the diagnostics subsystem's share of compiler source
(28.2% at the time of writing).

## Reproduce

```powershell
powershell -ExecutionPolicy Bypass -File tools\measure.ps1   # builds, generates corpus, collects CSVs
python tools\analyze.py                                      # writes experiments\out\results.md
git show --stat 48b5054                                      # E5 raw data
```

Outputs land in `experiments\out\` (gitignored; results tables are
copied into the paper, CSVs archived with the artifact).
